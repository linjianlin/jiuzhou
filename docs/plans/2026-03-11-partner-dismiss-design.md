# 伙伴下阵设计

**目标**

支持玩家主动让当前出战伙伴下阵，使角色可以在不携带任何伙伴的状态下进入战斗；同时在角色当前没有出战伙伴时，新获得伙伴默认保持未出战。

**核心方案**

1. 保持“同一角色最多只有 1 名出战伙伴”的现有数据模型，不新增新的出战位或编队结构。
2. 把伙伴出战状态更新收敛到共享模块，统一支持“设为某伙伴出战”和“清空当前出战伙伴”两种动作，避免服务层重复拼 SQL。
3. 后端新增单独的“下阵”接口，不复用 `/partner/activate` 传空参数，保证动作语义清晰。
4. 前端继续围绕 `activePartnerId: number | null` 渲染；当值为 `null` 时，明确表示角色当前未携带伙伴。
5. 伙伴获得链路取消“只要当前无出战伙伴就自动上阵”的旧规则，改为统一默认未出战，交由玩家手动控制。

**数据流**

1. 玩家在伙伴弹窗中查看伙伴列表或当前伙伴详情。
2. 若当前伙伴未出战，前端调用 `POST /partner/activate`，请求把该伙伴设为出战。
3. 若当前伙伴已出战，前端调用新的下阵接口，请求把当前出战状态清空。
4. 服务层通过共享出战状态更新入口改写 `character_partner.is_active`，然后重新返回最新伙伴总览片段。
5. 单人战斗开始时，战斗服务继续读取 `buildActivePartnerBattleMember()`；若当前没有出战伙伴，则返回 `null`，战斗中不带伙伴参战。
6. 新获得伙伴时，创建实例逻辑直接按未出战写入；总览刷新后若没有出战伙伴，前端仅默认选中列表第一个伙伴展示，不隐式代替玩家上阵。

**复用与抽象**

1. 后端扩展 `server/src/services/shared/partnerActivation.ts`，把“切到指定伙伴出战”和“清空出战伙伴”集中为单一入口，供 `partnerService.activate` 与新的 `partnerService.dismiss` 复用。
2. 前端复用现有伙伴总览刷新链路，不手动拼局部 `isActive` 状态，避免列表、详情卡片和战斗入口口径不一致。
3. 伙伴创建入口 `createPartnerInstanceFromDefinition` 继续作为唯一实例化入口，只调整其默认出战规则，避免主线奖励和招募确认分叉。

**边界条件**

1. 下阵动作不携带 `partnerId`，因为玩家同时只能有 1 名出战伙伴；服务层只需清空当前角色的出战标记。
2. 当角色本来就没有出战伙伴时，下阵接口应保持幂等，返回成功且 `activePartnerId = null`，避免前端为刷新态再补兜底逻辑。
3. 当前总览仍允许存在多个伙伴，但 `activePartnerId` 只能是某个伙伴 ID 或 `null`，不能出现第二种特殊状态。
4. 战斗侧不引入额外兼容分支；唯一数据源仍然是 `character_partner.is_active = true` 的查询结果。
5. 新获得伙伴默认未出战后，前端选中逻辑必须在 `activePartnerId` 为空时回退到列表第一个伙伴，保证弹窗仍有可见详情。

**非目标**

1. 不改造为多伙伴编队或助战位系统。
2. 不调整伙伴属性、技能、招募、升级、打书等已有业务规则。
3. 不新增“批量下场”“记忆上次出战伙伴”等扩展交互。
